# Open Questions

この文書では、仕様確認事項を以下の 3 分類で管理する。

- **回答済み**: Ver.1 の方針として決定済み。
- **Ver.1では仮定**: Ver.1 実装を進めるための暫定方針。必要に応じて後で見直す。
- **将来確認**: Ver.1 実装のブロッカーではないが、将来拡張前に確認する。

## 1. 回答済み

### 1.1 データ仕様

1. Ver.1 の主入力は AMASA `.d` ファイルとする。
2. Ver.1 の主対象は `.d` ファイル内の `S` event スケーラーデータとする。
3. `.root` ファイル、ROOT `TTree`、ROOT histogram、ROOT マクロ出力は Ver.1 の必須入力には含めない。
4. ch20 AS trigger は将来拡張扱いとし、Ver.1 では表示・監視対象に含めない。
5. `A` event のモニター表示は将来拡張扱いとし、Ver.1 では実装しない。
6. `S` event の ch1–ch7 / ch16–ch19 が行内に存在しない場合は、Ver.1 では欠損として扱う。

### 1.2 判定仕様

1. Dead / Recovery 判定は、Ver.1 では detector ch1–ch7 と Any ch16–ch19 の両方に適用する。
2. Dead / Recovery 判定結果は、Ver.1 ではまず別 JSON として出力する。
3. Web 表示では、可能ならグラフ上に帯またはマーカーとして表示する。
4. 表示が複雑になる場合は、Dead / Recovery / Missing Data の JSON 出力を優先する。
5. 欠損が 10 秒以上続いた場合は Dead とは別の Missing Data として扱う。
6. Dead / Recovery と Missing Data は別のイベント種別として扱う。

### 1.3 Web 表示

1. Ver.1 のトップページは最低限でよい。
2. 日付選択、チャンネル選択、グラフ表示、判定結果の確認ができれば十分とする。
3. 複数段表示は、まず detector ch1–ch7 と Any ch16–ch19 の 2 グループを優先する。
4. チャンネルごとの詳細表示は、可能なら同一ページ内で選択できる設計にする。
5. Ver.1 では日単位 JSON を基本とし、巨大な単一 JSON を作らない。
6. Plotly は Ver.1 では CDN 読み込みでよい。
7. Plotly は将来的に同梱版へ切り替えやすい構成にする。

### 1.4 運用

1. Ver.1 ではローカル出力ディレクトリに HTML / JSON を生成するところまでを優先する。
2. 公開先への転送処理は設定値で切り替えられる設計にする。
3. Ver.1 では手動実行できる CLI を先に作る。
4. cron 化は後続タスクにする。
5. run ファイル数が最大 72 に満たない日について、Ver.1 では alert しない。
6. run ファイル数は処理ログまたは summary JSON に記録する。
7. ログは `logs/` または `output/logs/` に日付別ログとして出す設計案を採用する。
8. 実データや不要に大きなログは Git 管理しない。

### 1.5 開発環境

1. Python はまず Python 3.10+ を想定する。
2. ROOT は Ver.1 の必須依存にはしない。
3. WSL2 Ubuntu では ROOT 非依存テストを基本にする。
4. 小さい合成または匿名化サンプルデータを `data/sample/` に追加してよい。
5. 本物の実験データや大きなファイルは追加しない。

## 2. Ver.1では仮定

1. ch21 と ch32 は意味未確定のため、Ver.1 では監視対象に含めない。
2. ROOT 入力の key 名、TTree 名、branch 名、単位、timezone は未確定だが、Ver.1 の `.d` 入力実装を止める理由にはしない。
3. 欠損は `0` ではなく missing value として扱い、Dead 判定に含めない。
4. Missing Data は Dead / Recovery と別イベントとして JSON に出力できる設計にする。
5. 日単位 JSON を基本とし、長期表示は複数の日単位 JSON または集約済み JSON を読み分ける設計にする。
6. 公開先への転送は Ver.1 の必須実行ステップにせず、ローカル出力を優先する。
